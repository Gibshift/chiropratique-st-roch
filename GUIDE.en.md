## 1. Running the project locally

### Prerequisites
- Node.js 18+ installed
- `pnpm` installed (`npm install -g pnpm`)

### Steps

**1. Clone and install dependencies**
```bash
git clone <repo-url>
cd chiropratique-st-roch
pnpm install
```

**2. Create the `.env` file**  
Copy `.env.example` to `.env` and fill in the variables:

```env
# PostgreSQL database (Neon)
DATABASE_URL=postgresql://...

# Payload CMS — secret key (any long random string)
PAYLOAD_SECRET=some-long-random-string

# Media storage (Cloudflare R2)
R2_BUCKET=...
R2_ENDPOINT=...
R2_ACCESS_KEY_ID=...
R2_SECRET_ACCESS_KEY=...

# Public site URL in development
NEXT_PUBLIC_SERVER_URL=http://localhost:3000

# Optional — automatic SEO generation via Claude AI
ANTHROPIC_API_KEY=...
```

> **Why so many variables?** This project connects several external services: the database is hosted on Neon (cloud PostgreSQL), images are stored on Cloudflare R2, and SEO text is generated by the Claude API.

**3. Start the development server**
```bash
pnpm dev
```

The site is now available at `http://localhost:3000`.  
The Payload admin panel is at `http://localhost:3000/admin`.

> **First run?** On the first startup, Payload automatically creates all the database tables. You'll see migration messages in the terminal — that's normal.

**4. Create an admin account**  
Go to `http://localhost:3000/admin` and create your account. The first time, anyone can create an account (after that, only an admin can create more).

## 2. Architecture at a glance

```
src/
├── app/
│   ├── (frontend)/        ← Public pages (what visitors see)
│   └── (payload)/         ← Admin interface + API (what editors see)
├── collections/           ← Data schemas (Services, Posts, Professionals...)
├── globals/               ← Site-wide data (SiteSettings)
├── Header/                ← Site header (navigation, info bar)
├── Footer/                ← Site footer
├── components/            ← All React visual components
│   ├── home/              ← Home page
│   ├── services/          ← Service pages
│   ├── conditions/        ← Treated conditions pages
│   ├── professionals/     ← Professionals pages
│   ├── blogue/            ← Blog pages
│   ├── contact/           ← Contact page
│   └── ui/                ← Small reusable components
├── utilities/             ← Utility functions (slugify, Jane URLs, etc.)
├── plugins/               ← Payload plugin configuration
└── blocks/                ← CMS content blocks (Archive, CTA, Media...)
```

### How data flows

```
Visitor goes to /services
    ↓
app/(frontend)/services/page.tsx    ← thin Next.js route file
    ↓
components/services/ServicesPage.tsx ← server component that reads the DB
    ↓
Renders the page with data
```

> **Why this separation?** The `page.tsx` file is intentionally kept thin — it only does two things: configure SEO metadata and call the component. All logic lives in `components/`.

## 4. How to edit a page

### Editing text or an image on a static page (e.g. contact, home)

→ Edit the file directly in `src/components/{section}/`.  
Example: to change the contact page text → `src/components/contact/ContactPage.tsx`.

### Editing the content of a service, professional, or blog post

→ Use the **Payload admin** (`/admin`). No code changes needed.  
Pages update automatically after saving.

> **General rule**: content (text, images, data) → Payload admin. Layout and design → code.

### Changing the browser tab title

→ `src/app/(frontend)/layout.tsx`, in the `metadata` export:
```ts
title: {
  default: 'Chiropratique St-Roch | Clinique à Québec',
  template: '%s | Chiropratique St-Roch',
},
```

### Adding a new CMS page (no code required)

1. In the Payload admin → **Pages** → Create a new document
2. Set a slug (e.g. `about-us`)
3. Add a hero and content blocks
4. The page is automatically available at `/{slug}` — no code needed

### Adding a new custom page (React code)

1. Create a folder at `app/(frontend)/my-page/`
2. Create `page.tsx` inside it (copy an existing `page.tsx` as a template)
3. Create the component at `src/components/my-section/MyPage.tsx`
4. Add the link to the header/footer via the Payload admin

### Editing a page's SEO metadata

Each `[slug]/page.tsx` has a `generateMetadata` function. This is where the title, description, and Open Graph images (for social sharing) are built.

> **Note**: some pages use `title: { absolute: '...' }` to prevent the layout template (`| Chiropratique St-Roch`) from being appended twice.

## 5. Payload CMS — collections and globals

Payload CMS is the site's admin interface. It manages content and the database.

### Collections (content items)

| Collection | What it is | Key fields |
|---|---|---|
| **Services** | The 5 clinic disciplines | `title`, `slug`, `shortDescription`, `isFeatured` (show on home), `order` (display order) |
| **Conditions** | Treated conditions (e.g. back pain) | `title`, `category` (body region), `isFeatured`, `order` |
| **Condition Categories** | Condition groups (e.g. "Spine") | `title`, `slug`, `conditionsList`, `relatedServices` |
| **Professionals** | Team members | `name`, `title`, `shortBio`, `photo`, `isActive` (hide without deleting) |
| **Posts** | Blog articles | `title`, `content`, `heroImage`, `publishedAt`, `categories` |
| **Pages** | Generic pages with blocks | `title`, `slug`, `hero`, `layout` |
| **Media** | Uploaded images and files | `alt`, stored on Cloudflare R2 |

### Globals (site-wide data)

| Global | What it controls |
|---|---|
| **Header** | Navigation menu links |
| **Footer** | Footer navigation links |
| **Site Settings** (SiteSettings) | Phone, address, opening hours, booking URL, social media |

> **Important**: to change the phone number, address, hours, or booking URL → Payload admin → **Site Settings**. No code needed.

## 6. Global data (Header, Footer, SiteSettings)

**SiteSettings is the single source of truth** for all clinic information. It is read by:
- The header (phone, address, hours)
- The footer (contact info, social media)
- The contact page
- Schema.org structured data (local SEO)

**For Jane App booking links** → the fallback lives in `src/utilities/jane.ts`:
```ts
export const JANE_BASE_URL = 'https://chiropratiquestroch.janeapp.com'
export const JANE_SERVICE_URLS = { ... }   // direct links by discipline
```
This is the **only place** to update if the Jane URL changes and SiteSettings isn't enough.

**Caching**: Header and Footer use a cache that automatically revalidates after each admin save. SiteSettings is read without a cache.

## 7. Images and media

There are two types of images in the project — don't mix them up.

### Type 1: Images uploaded via the admin (Payload media)

Stored on Cloudflare R2. Automatically converted to WebP and resized into 7 formats.

```tsx
// Retrieved from the DB, served directly via their URL
<img src={professional.photo?.url} alt={professional.name} />

// For thumbnails, use an appropriate size
<img
  src={professional.photo?.sizes?.medium?.url ?? professional.photo?.url}
  alt={professional.name}
/>
```

The 7 available sizes: `thumbnail` (300px), `square` (500×500), `small` (600px), `medium` (900px), `large` (1400px), `xlarge` (1920px), `og` (1200×630 for social sharing).

> These images do **not** go through Next.js `<Image>` optimization — they're already optimized by Payload.

### Type 2: Static site assets (`public/assets/`)

PNG illustrations, silhouettes, icons. Referenced with an absolute path.

```tsx
// Always use next/image for static assets → automatic WebP + lazy loading
import Image from 'next/image'
<Image src="/assets/salle-chiro.png" alt="Clinic room" width={800} height={600} />

// Hero image that needs to be visible immediately → add priority to avoid slow LCP
<Image src="/assets/hero.png" alt="..." fill sizes="100vw" priority />
```

## 8. CSS and styles

The project uses **Tailwind CSS v4** — styles are utility classes written directly in JSX.

### Fonts

| Variable | Font | Used for |
|---|---|---|
| `var(--font-dm-sans)` | DM Sans | Body text, paragraphs |
| `var(--font-barlow-condensed)` | Barlow Condensed | Headings, labels, nav, buttons |
| `var(--font-geist-mono)` | Geist Mono | Code (inherited from the template) |

Fonts are self-hosted (via `next/font/google` in `layout.tsx`) — no external requests to Google Fonts.

### Color palette

| Role | Tailwind class | Context |
|---|---|---|
| Clinic red | `text-red-700` | On white or cream background |
| Red on dark background | `text-red-500` | Footer, dark sections |
| Dark background | `bg-zinc-950` | Footer, mobile drawer |
| Cream background | `bg-[#f6f1e8]` | Some home page sections |
| Main text | `text-zinc-950` | Body text |
| Secondary text | `text-zinc-600` | Dates, metadata |

> **WCAG AA rule**: on cream background `#f6f1e8`, use `red-700` (not `red-600`). On dark background (`zinc-950`), use `red-500` or `zinc-300`.

### Scroll animations

The `<ScrollReveal>` component (in `ui/ScrollReveal.tsx`) uses `IntersectionObserver` to trigger fade-in animations when elements enter the viewport. It transitions elements from `sr-hidden` to `sr-visible`. Styles are in `globals.css`.

## 10. Known duplications — watch out when editing

These patterns exist in multiple places in the code. If you change one, check the others.

### Professional photo extraction pattern

This block is copied across 3 components:
```ts
const photoUrl =
  professional.photo &&
  typeof professional.photo === 'object' &&
  'url' in professional.photo
    ? professional.photo.url
    : null
```
Present in: `ClinicHomePage.tsx`, `ProfessionalsPage.tsx`, `ServiceDetailPage.tsx`.

### Page hero sections (Services and Professionals)

`ServicesPage.tsx` and `ProfessionalsPage.tsx` each implement their own inline hero instead of sharing a `<PageHero>` component. This is intentional — the designs are too different to abstract (cream vs white background, different dimensions).

### Daily article rotation logic

The `dayIndex = Math.floor(Date.now() / 86400000)` logic (rotating the displayed article each day) is duplicated in `ClinicHomePage.tsx` and `BloguePage.tsx`.

## 11. Project conventions

### File structure
- **`page.tsx` is always thin**: it exports `generateMetadata` and a `Page` function that calls the component — nothing else.
- **Client components** (with React state, events): suffixed `.client.tsx`
- **Server components** (direct DB reads): no suffix

### Naming
- Route folders: `kebab-case` (`conditions-traitees/`, `[slug]/`)
- Component files: `PascalCase` (`ServicesPage.tsx`)
- Utility files: `camelCase` (`getGlobals.ts`)
- Source code: English. Payload admin labels: French.

### Slugs
Auto-generated by `utilities/slugify.ts` via a Payload `beforeValidate` hook. Accents are normalized (`é` → `e`, `ô` → `o`). Never enter a slug with uppercase letters or spaces.

### SEO metadata
- `siteName`: always `'Chiropratique St-Roch'`
- `locale`: always `'fr_CA'`
- To prevent the title template from being appended twice: use `title: { absolute: '...' }`
- All OG images go through `getDefaultOpenGraphImages()` or `getOpenGraphImages()`

### Automatic SEO generation (Claude AI)

On every admin save, the plugin in `src/plugins/index.ts` calls the Anthropic API (Haiku model) to generate an SEO title and meta description. Works for: posts, services, condition-categories, professionals. Requires `ANTHROPIC_API_KEY` in `.env`. If the key is absent, the SEO field stays empty — no error is thrown.

### Next.js cache revalidation

| What | How it revalidates |
|---|---|
| Header and Footer | Automatically via their Payload `afterChange` hook |
| Pages and Posts | `revalidatePage` / `revalidatePost` hooks in their collections |
| Home page and blog | `export const revalidate = 3600` (re-rendered every hour) |

### Payload and TypeScript

Some `as any` casts in the code are intentional:
- `collection: 'condition-categories' as any` in `payload.find()` calls — this custom slug isn't in the default generated types
- `doc as { id: ...; title: string }` in `beforeSync.ts` — the partial `select` makes the return type incomplete

## 12. Glossary for beginners

**Payload CMS**  
The system that manages the site's content (articles, services, professionals). It's a "headless CMS": it stores data in a PostgreSQL database and exposes it via an API, but doesn't impose any design.

**Next.js App Router**  
The React framework that powers the site. The App Router (since Next.js 13) organizes pages in folders with `page.tsx` files. The parentheses in `(frontend)` and `(payload)` folder names are "route groups" — they organize the code without affecting URLs.

**Server component vs client component**  
- **Server**: runs only on the server, can read the database directly, no interactivity. Files without `'use client'`.  
- **Client**: runs in the browser, can use `useState`, `useEffect`, events. Files with `'use client'` as the first line.

**Drizzle ORM**  
The library Payload uses internally to talk to PostgreSQL. You don't interact with Drizzle directly — Payload handles it. The `push: true` in `payload.config.ts` makes Payload automatically sync the database schema on startup.

**Lexical**  
The rich text editor used by Payload for article content. It stores content as JSON (not HTML). The `RichText` component in `blocks/` converts it to HTML for display.

**Cloudflare R2**  
The image storage service (similar to Amazon S3). When you upload an image in Payload, it's sent to R2 — not stored on the Next.js server.

**ISR (Incremental Static Regeneration)**  
The Next.js technique that allows pages to be pre-generated statically but refreshed periodically (via `revalidate`) or on demand (via `revalidatePath`). The home page refreshes every hour.

**OG (Open Graph)**  
The HTML tags that control how a page appears when shared on Facebook, LinkedIn, iMessage, etc. — title, description, preview image.

**Slug**  
The part of a URL that identifies a specific page. For `/services/chiropratique`, the slug is `chiropratique`. Always lowercase, no accents, no spaces.

**Schema.org / JSON-LD**  
Structured data readable by Google to improve appearance in search results (opening hours, address, contact info). Managed by `components/seo/LocalBusinessJsonLd.tsx`.
